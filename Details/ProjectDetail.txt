═══════════════════════════════════════════════════════════════════════════════

Collabzy is a web-based influencer marketing platform designed to bridge the gap 
between social media influencers and brands seeking promotional collaborations. 
The existing influencer marketing landscape relies heavily on informal 
communication through social media direct messages, leading to inefficiencies, 
lack of transparency, and low accountability. This project addresses these 
challenges by implementing a structured, professional platform that facilitates 
direct connections between influencers and brands.

The platform has been developed using modern web technologies including React for 
the frontend, Node.js with Express for the backend, MongoDB for data storage, and 
Socket.io for real-time communication features. The system implements role-based 
access control to support three user types: Influencers (Content Creators), Brands 
(Product Promoters), and Administrators (Platform Moderators).

As of April 2026, the project has achieved 80% completion with backend development 
fully completed (100%), frontend UI fully built (100%), and frontend-backend 
integration at 85% completion. The project encompasses comprehensive features 
including user authentication, profile management, collaboration request handling, 
real-time messaging, notification systems, payment tracking, and administrative 
monitoring capabilities.

Keywords: Influencer Marketing, Digital Platform, Real-time Communication, 
Web Application, Marketplace

6.1 Background and Context

Social media has revolutionized digital marketing, with influencer marketing 
emerging as one of the most effective promotional strategies. However, the current 
influencer marketing ecosystem operates in a fragmented manner, relying primarily 
on informal communication channels.

6.2 Problem Statement

Current challenges in influencer marketing include:

• Difficulty in Discovery: Brands struggle to find influencers suitable for their 
  niche products and target audience.

• Scattered Communication: Collaboration negotiations happen through multiple 
  unorganized social media channels.

• Lack of Transparency: No clear system for defining deliverables, timelines, and 
  pricing structures.

• Trust Issues: Absence of accountability mechanisms leading to potential fraud 
  and unfulfilled collaborations.

• No Tracking: Progress of collaborations cannot be systematically monitored.

6.3 Project Motivation

Collabzy aims to professionalize influencer marketing by creating a dedicated 
platform that:

• Centralizes all collaboration processes in one place
• Implements transparent communication and tracking mechanisms
• Establishes trust through accountability and review systems
• Reduces time wasted in finding and negotiating collaborations
• Provides data-driven insights for both influencers and brands

6.4 Project Objectives

The primary objectives of Collabzy are:

1. To create a digital marketplace connecting influencers and brands
2. To reduce manual effort in finding and managing collaborations
3. To provide transparency in pricing and service offerings
4. To establish trust and accountability through structured collaboration tracking
5. To enable real-time communication between stakeholders
6. To implement administrative oversight for platform security and fair use

7.1 Current Market Solutions

Existing influencer marketing approaches include:

7.1.1 Social Media Direct Messaging
- Method: Brands contact influencers directly via Instagram, YouTube DMs
- Advantages: Direct communication, no intermediary
- Disadvantages: Unorganized, no contract formalization, poor tracking

7.1.2 Influencer Agencies
- Method: Third-party agencies match influencers with brands
- Advantages: Professional management, reduced fraud
- Disadvantages: High commission fees (20-40%), less control for both parties

7.1.3 General Freelance Platforms (Upwork, Fiverr)
- Method: Influencers post services on marketplace
- Advantages: Some structure and accountability
- Disadvantages: Designed for general services, not influencer-specific

7.2 Technology Stack Adopted

- React.js: Component-based UI framework for interactive user interfaces

Features Developed:

7.2.2 Backend Technology

- Regular testing and feedback incorporation

7.3.2 MVC Architecture

7.3.3 Security Practices

- Limited influencer choice and control
- Centralized power with agencies rather than distributed marketplace
- Slow negotiation processes
- Not accessible to independent creators or small brands

8.4 General Marketplace Limitations
- Lack of platform-specific metrics and insights
- No formal collaboration tracking system
- Missing features for deal management and progress tracking
- Limited communication capabilities
- No built-in review and rating system specific to influencer work

10.1 Overall Architecture

Collabzy follows a three-tier client-server architecture:

┌─────────────────────────────────────────────────────────────────┐
│                    CLIENT LAYER (Frontend)                       │
│                      React Application                           │
│         (Components, Pages, State Management, Views)             │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                      HTTP/WebSocket
                           │
┌──────────────────────────┴──────────────────────────────────────┐
│                 APPLICATION LAYER (Backend)                      │
│        Node.js + Express Server with Business Logic              │
│  (Routes, Controllers, Services, Middleware, Authentication)     │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                      MongoDB Driver
                           │
┌──────────────────────────┴──────────────────────────────────────┐
│                    DATA LAYER (Database)                         │
│                MongoDB Database Collections                      │
│             (Users, Profiles, Campaigns, Messages, etc.)         │
└─────────────────────────────────────────────────────────────────┘

10.3 Backend Architecture

Technology: Node.js + Express.js + MongoDB

Directory Structure:
```
backend/
├── config/              # Configuration files (Database, Security, Socket)
├── controllers/         # Business logic for each feature
├── models/             # MongoDB schemas and data models
├── routes/             # API endpoint definitions
├── middleware/         # Custom middleware (Auth, Validation)
├── services/           # Utility services (Payment, Notification, Filtering)
├── constants/          # Application constants
├── utils/              # Helper functions
└── server.js           # Application entry point
```

API Endpoints (110+ Endpoints):
- Authentication: /api/auth/* (Register, Login, Logout, Password Reset)
- Profiles: /api/influencer/*, /api/brand/* (CRUD operations)
- Campaigns: /api/campaigns/* (Create, List, Detail, Update)
- Applications: /api/applications/* (Apply, Review, Accept, Reject)
- Deals: /api/deals/* (Create, Update Status, Track Progress)
- Messages: /api/messages/* (Send, Retrieve, Mark Read)
- Notifications: /api/notifications/* (Get, Mark Read)
- Reviews: /api/reviews/* (Create, List, Update)
- Admin: /api/admin/* (Manage Users, Reports, Content Moderation)

11.1 Data Models

11.1.1 User Schema
Fields:
- userId (Primary Key): Unique identifier
- email: User's email address (unique)
- password: Hashed password
- username: Display name
- role: Account type (influencer, brand, admin)
- profilePicture: Profile image URL
- createdAt: Account creation timestamp
- updatedAt: Last update timestamp
- isVerified: Email verification status
- accountStatus: Active/Inactive/Suspended

11.1.2 InfluencerProfile Schema
Fields:
- profileId (Primary Key): Unique identifier
- userId (Foreign Key): Reference to User
- bio: Profile description
- niches: Array of categories (Fashion, Tech, Beauty, etc.)
- followers: Number of social media followers
- engagementRate: Average engagement percentage
- instagramHandle: Social media username
- youtubeHandle: Social media username
- portfolioLinks: Array of past work examples
- rates: Pricing structure
- availability: Current availability status
- ratingScore: Average rating (1-5)
- totalCollaborations: Number of completed deals
- verificationStatus: Verified/Unverified

11.1.3 BrandProfile Schema
Fields:
- profileId (Primary Key): Unique identifier
- userId (Foreign Key): Reference to User
- brandName: Official brand name
- description: Brand description
- industry: Industry category
- website: Brand website URL
- logo: Brand logo URL
- budget: Marketing budget indicator
- preferredNiches: Target influencer niches
- rating: Brand rating from influencers (1-5)
- totalCampaigns: Number of campaigns launched
- verificationStatus: Verified/Unverified

11.1.4 Campaign Schema
Fields:
- campaignId (Primary Key): Unique identifier
- brandId (Foreign Key): Reference to BrandProfile
- title: Campaign name
- description: Campaign details and requirements
- targetNiches: Influencer categories needed
- budget: Campaign budget
- deliverables: List of required deliverables
- timeline: Start and end dates
- status: Draft/Active/Completed/Cancelled
- visibilityScope: Public/Private
- createdAt: Campaign creation date
- updatedAt: Last modification date
- applicantCount: Total applications received

11.1.5 Application Schema
Fields:
- applicationId (Primary Key): Unique identifier
- campaignId (Foreign Key): Reference to Campaign
- influencerId (Foreign Key): Reference to InfluencerProfile
- brandId (Foreign Key): Reference to BrandProfile
- appliedAt: Application submission date
- status: Pending/Accepted/Rejected/Withdrawn
- proposedRate: Influencer's proposed rate
- message: Application cover letter
- reviewedAt: When decision was made
- rejectionReason: Reason if rejected

11.1.6 Deal Schema
Fields:
- dealId (Primary Key): Unique identifier
- applicationId (Foreign Key): Reference to Application
- influencerId (Foreign Key): Reference to InfluencerProfile
- brandId (Foreign Key): Reference to BrandProfile
- campaignId (Foreign Key): Reference to Campaign
- agreedRate: Finalized compensation
- status: Pending/InProgress/Completed/Cancelled
- deliveryDate: Expected delivery date
- actualDeliveryDate: When delivered
- notes: Special terms and conditions
- createdAt: Deal creation date
- completionPercentage: Progress tracking (0-100%)

11.1.7 Message Schema
Fields:
- messageId (Primary Key): Unique identifier
- senderId (Foreign Key): Reference to User
- receiverId (Foreign Key): Reference to User
- dealId (Foreign Key): Reference to Deal (null if general chat)
- content: Message text
- isRead: Read status indicator
- readAt: Timestamp when read
- sentAt: Message send timestamp
- attachments: File or image URLs (if any)

11.1.8 Notification Schema
Fields:
- notificationId (Primary Key): Unique identifier
- userId (Foreign Key): Reference to User
- type: Notification type (application, message, deal, review, etc.)
- title: Notification headline
- message: Notification content
- relatedEntityId: ID of related object (campaignId, dealId, etc.)
- isRead: Read status
- createdAt: Notification creation time
- actionUrl: Link to related page

11.1.9 Review Schema
Fields:
- reviewId (Primary Key): Unique identifier
- dealId (Foreign Key): Reference to Deal
- reviewerId (Foreign Key): Reference to User
- recipientId (Foreign Key): Reference to User
- rating: Numeric rating (1-5)
- comment: Written review text
- categories: Ratings for specific aspects
- createdAt: Review creation date
- helpful: Count of helpful votes

11.1.10 Payment Schema
Fields:
- paymentId (Primary Key): Unique identifier
- dealId (Foreign Key): Reference to Deal
- payerId (Foreign Key): Reference to User (Brand)
- payeeId (Foreign Key): Reference to User (Influencer)
- amount: Payment amount
- status: Pending/Completed/Failed/Refunded
- method: Payment method (bank, wallet, card)
- transactionId: External transaction reference
- createdAt: Payment request date
- completedAt: Payment completion date
- invoiceUrl: Invoice document link

11.1.11 Report Schema
Fields:
- reportId (Primary Key): Unique identifier
- reporterId (Foreign Key): Reference to User
- reportedUserId (Foreign Key): Reference to User
- reportType: Reason for report
- description: Report details
- evidence: Supporting files/screenshots
- status: Pending/Reviewing/Resolved/Dismissed
- createdAt: Report submission date
- adminNotes: Investigation notes

11.1.12 Violation Schema
Fields:
- violationId (Primary Key): Unique identifier
- userId (Foreign Key): Reference to User
- violationType: Type of policy violation
- severity: Low/Medium/High/Critical
- description: Violation description
- evidence: Proof/documentation
- action: Warning/Suspension/Ban
- createdAt: Violation record date
- resolvedAt: When action was taken

11.2 Data Relationships

User (1) ─── (Many) InfluencerProfile / BrandProfile
User (1) ─── (Many) Message (as sender or receiver)
BrandProfile (1) ─── (Many) Campaign
BrandProfile (1) ─── (Many) Deal (as brand party)
Campaign (1) ─── (Many) Application
InfluencerProfile (1) ─── (Many) Application
Application (1) ─── (1) Deal
Campaign (1) ─── (Many) Deal
User (1) ─── (Many) Notification
User (1) ─── (Many) Review (as reviewer or reviewee)
Deal (1) ─── (1) Payment
User (1) ─── (Many) Report (as reporter or reported)
User (1) ─── (Many) Violation

11.3 Data Validation Rules

- Email: Must be unique and valid email format
- Password: Minimum 8 characters, alphanumeric
- Rating: Must be between 1 and 5
- Budget: Must be positive number
- Phone (if used): Must be valid format
- Dates: Must be logical (end date after start date)
- Status Fields: Must match predefined enums

11.4 Indexing Strategy

Indexes created for:
- Email (User) - Unique index for login
- UserId (All related documents) - Foreign key indexing
- CampaignId (Application, Deal) - Frequent query filter
- Status Fields - Query optimization
- CreatedAt (Messages, Notifications) - Timestamp-based sorting
- CombinedIndexes (dealId, influencerId) for complex queries
